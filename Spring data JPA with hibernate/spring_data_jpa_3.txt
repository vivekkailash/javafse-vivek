Hands on 1 - HQL and JPQL intro
-------------------------------

** just notes, no code **

- HQL = Hibernate Query Language
- JPQL = Java Persistence Query Language
- both are object based (use class names, not table names)
- JPQL is a subset of HQL - every JPQL is valid HQL, not the other way
- both support SELECT, UPDATE, DELETE
- HQL also allows INSERT (extra)

Example: FROM Employee e WHERE e.permanent = 1
   -> Employee is the CLASS name, not table
   -> permanent is the FIELD name, not column



Hands on 2 - Get all permanent employees using HQL
--------------------------------------------------

EmployeeRepository.java (add method with @Query)
------------------------------------------------
package com.cognizant.ormlearn.repository;

import java.util.List;

import org.springframework.data.jpa.repository.JpaRepository;
import org.springframework.data.jpa.repository.Query;
import org.springframework.stereotype.Repository;

import com.cognizant.ormlearn.model.Employee;

@Repository
public interface EmployeeRepository extends JpaRepository<Employee, Integer> {

    @Query(value="SELECT e FROM Employee e WHERE e.permanent = 1")
    List<Employee> getAllPermanentEmployees();

    // NOTE: HQL - "Employee" is the class not the table
}


EmployeeService.java (add method)
---------------------------------
@Transactional
public List<Employee> getAllPermanentEmployees() {
    return employeeRepository.getAllPermanentEmployees();
}

    employees.forEach(e -> LOGGER.debug("Skills: {}", e.getSkillList()));

Skills: [Skill[id=1,name=Java], Skill[id=2,name=SQL]]

** problem: multiple queries fired (N+1 issue) **


Fix 1 - Remove eager fetch, use join fetch in HQL
-------------------------------------------------

Department.java   -> remove fetch=EAGER from @OneToMany employeeList
Employee.java     -> remove fetch=EAGER from @ManyToMany skillList

Then update the query in EmployeeRepository:

@Query(value="SELECT e FROM Employee e "
           + "left join fetch e.department d "
           + "left join fetch e.skillList "
           + "WHERE e.permanent = 1")
List<Employee> getAllPermanentEmployees();


** now only ONE query runs. big improvement **


Output (same result but single SQL fired):
Permanent Employees: [Employee[id=1,...], Employee[id=2,...], Employee[id=4,...]]
Skills: [Skill[id=1,name=Java], Skill[id=2,name=SQL]]
Skills: [Skill[id=3,name=Angular]]
Skills: [Skill[id=1,name=Java]]


** takeaway - "join" links tables, "fetch" populates the beans. **
** if you don't use fetch, the association won't be loaded (lazy) **

AttemptRepository.java
----------------------
@Repository
public interface AttemptRepository extends JpaRepository<Attempt, Integer> {

    @Query(value = "SELECT a FROM Attempt a "
                 + "join fetch a.user u "
                 + "join fetch a.attemptQuestions aq "
                 + "join fetch aq.question q "
                 + "join fetch aq.attemptOptions ao "
                 + "join fetch ao.option o "
                 + "WHERE u.id = :userId AND a.id = :attemptId")
    Attempt getAttempt(@Param("userId") int userId,
                       @Param("attemptId") int attemptId);
}


AttemptService.java

@Service
public class AttemptService {

    @Autowired
    private AttemptRepository attemptRepository;

    @Transactional
    public Attempt getAttempt(int userId, int attemptId) {
        return attemptRepository.getAttempt(userId, attemptId);
    }
}

Hands on 6 - Criteria Query
---------------------------

** scenario: amazon-like search with many optional filters **
** if we use fixed HQL, we can't handle dynamic where clauses well **
** Criteria Query = build query programmatically at runtime **


Example code (just for reference, not required to run):
-------------------------------------------------------

@Autowired
private EntityManager entityManager;

public List<Employee> searchEmployees(String name, Double minSalary, Boolean permanent) {

    CriteriaBuilder cb = entityManager.getCriteriaBuilder();
    CriteriaQuery<Employee> cq = cb.createQuery(Employee.class);
    Root<Employee> root = cq.from(Employee.class);

    List<Predicate> predicates = new ArrayList<>();

    // add filters only if user gave them
    if (name != null) {
        predicates.add(cb.like(root.get("name"), "%" + name + "%"));
    }
    if (minSalary != null) {
        predicates.add(cb.greaterThan(root.get("salary"), minSalary));
    }
    if (permanent != null) {
        predicates.add(cb.equal(root.get("permanent"), permanent));
    }

    cq.where(predicates.toArray(new Predicate[0]));

    TypedQuery<Employee> query = entityManager.createQuery(cq);
    return query.getResultList();
}


Key classes:
- CriteriaBuilder  -> builds pieces (like, equal, greaterThan)
- CriteriaQuery    -> the whole query
- Root             -> the "from" table
- TypedQuery       -> the runnable query object


Output (call with name="vi", minSalary=25000, permanent=true):
Employee [id=5, name=Vivek, salary=30000.0, permanent=true, ...]


** big win: we only add filters that the user actually gave **
** no need to write 8 different fixed queries for each combo **


ref: https://howtodoinjava.com/hibernate/hibernate-criteria-queries-tutorial/
